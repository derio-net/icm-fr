Interpretable Context Methodology

Folder structure as agent architecture. One agent reads the right files at the right moment. The folder structure replaces the framework.

Van Clief & McDermott, Eduba / University of Edinburgh, 2025

The core idea

Framework approach
Multiple specialised agents coordinated through code. Change a step = edit code, redeploy. Inspect state = build a dashboard.
→
ICM approach
One agent reads different context at each stage. Change a step = rename a folder. Inspect state = open the folder, read the files.

Five principles

1
One stage, one job. Each folder handles one step. A stage that fetches data does not also filter it.
2
Plain text as the interface. Stages communicate through markdown and JSON. Any tool that can read a text file can participate.
3
Layered context loading. Agents load only the context they need for the current stage. Less irrelevant context = better output.
4
Every output is an edit surface. The human can open, read, edit, and save any intermediate output before the next stage runs.
5
Configure the factory, not the product. Set up once with your preferences. Each run produces a new deliverable using the same configuration.

Each layer answers one question. The agent loads layers top to bottom, but only loads what the current stage needs.

Where humans intervene

Observed across 33 practitioners: heavy editing at stage 1 (direction-setting), light in the middle (constrained execution), heavy again at the end (alignment checking). The middle stages get the lightest touch because both anchors are in place.

Where this works and where it does not

Works
  • Sequential workflows (step 2 follows step 1)
  • Human reviews output at each step
  • Same pipeline runs repeatedly with different input
  • Content production, research, training material
  • Non-developers need to make changes
Does not work
  • Real-time multi-agent collaboration
  • High-concurrency (many users at once)
  • Complex branching based on AI decisions
  • Dynamic agent-to-agent communication
  • Systems that need message-passing infrastructure

The claim is not that this replaces frameworks across the board. The claim is that for a large and common class of workflows, the existing tools provide more complexity than the problem requires. The simplest viable architecture for this class of problem is one that already exists on every computer: the filesystem.
